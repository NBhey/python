### Обработка исключений — детализация пройденного:
- [x] **Исключения как второй канал связи** между уровнями кода (помимо `return`)
- [x] **Граница доверия**: данные извне (`input`, файлы, сеть) опасны; данные изнутри программы безопасны
- [x] **EAFP** ("Easier to Ask Forgiveness than Permission") — питонячий стиль, для непредсказуемых ошибок извне
- [x] **LBYL** ("Look Before You Leap") — для предсказуемых бизнес-условий на своих данных
- [x] Базовый синтаксис: `try: ... except T: ...`
- [x] `except T as e:` — доступ к объекту исключения
- [x] `except (T1, T2):` — несколько типов одним блоком (теоретически, на практике не применяли)
- [x] `else:` — код, выполняемый только при успехе `try` (минимизация зоны покрытия)
- [x] `finally:` — код, выполняемый всегда (для уборки ресурсов)
- [x] `raise SomeError('message')` — генерация своих исключений
- [x] **Иерархия исключений**: `BaseException > KeyboardInterrupt, SystemExit, Exception > ValueError, KeyError, ...`
- [x] **Naked `except:` — антипаттерн** (ловит Ctrl+C → программа неубиваемая). Эксперимент с реальным Ctrl+C в PyCharm подтвердил это вживую.
- [x] `except Exception:` — допустимо как «всё, кроме служебного» (не ловит `KeyboardInterrupt`/`SystemExit`)
- [x] **Не лови, если не знаешь что делать** — пробрасывай наверх, к тому, кто знает
- [x] **Цель `except` — вернуть в рабочее состояние, а не заглушить ошибку**
- [x] **Проверяй причину, а не следствие**: `if not expenses`, а не `if all_total == 0`
- [x] **Согласованность обработки пустоты** — если обработал в одном месте, обработай везде
- [x] **Trace-back читается снизу вверх** — внизу источник проблемы, выше — путь вызовов
- [x] Конкретные типы исключений и когда какой летит:
  - `ValueError` — значение не подходит (`int("abc")`)
  - `TypeError` — тип не подходит (`"abc" + 5`)
  - `KeyError` — ключа нет в словаре
  - `IndexError` — индекса нет в списке
  - `ZeroDivisionError` — деление на ноль
  - `FileNotFoundError` — файла нет (будет в следующей теме)
  - `AttributeError` — нет атрибута/метода у объекта
- [x] **Retry-loop паттерн**: `while True: try: return ...; except T: print(...)`
- [x] **Применено в проекте**: `try/except ValueError` на `int(input(...))`; LBYL `if not expenses` вместо EAFP на `ZeroDivisionError`
- [x] Параллели с JS: try/catch в Python типизирован по умолчанию, в JS — нет; ошибки на границах (fetch/JSON.parse) ≈ try/except на input/файлах; `if data.length > 0 && <Component/>` ≈ LBYL на пустоте

- **`if not x` vs `if len(x) == 0`** — пока пишет вторым способом (нормально, но менее идиоматично). Постепенно переходим на питонячий стиль.

- **Звёздочка `*` в сигнатуре** — поначалу ставил «потому что так в примере было», без понимания зачем. Сейчас понимает: для функций с 3+ параметрами или фильтрами/опциями — да, для функций с одним аргументом — нет. Стоит периодически проверять.
- **Параметризация generic-типов** — пишет `list`, `tuple`, `dict` вместо `list[dict]`, `tuple[int, int]`. Не критично, но напоминать.
- **Docstring отстаёт от кода** — после рефакторинга функция меняется, а docstring остаётся старым. «Документация обновляется в той же коммите, что код».
- **Опечатки в русских docstrings и переменных** — мелкие, но регулярные («соответствющих», `formated_expenses` вместо `formatted_expenses`). Не критично, но глаз привыкает к опечатке.
- **Перенос логики при рефакторинге** — был случай: попросил «один проход» в `analyze_expenses`, он внутри вызвал `filter_expenses` (что не один проход) и пошёл циклом по `matched_expenses` вместо `expenses` → словил баг с `all_total`. Урок усвоен: «один проход» = делать всю работу внутри одного цикла по полному списку, не дёргать другие функции, которые тоже идут по списку.
- **🆕 Регрессии при рефакторинге сигнатуры** — переписав `filter_expenses` в одну строку через comprehension, потерял условие `category is None or ...` (стало просто `==`). Это типичный случай "чинил одно — сломал другое". Исправил быстро, но привычка проверять, что **поведение не изменилось**, ещё формируется.
- **🆕 Дублирование вычислений** — в первой версии финального рефакторинга вызвал `sum(...)` дважды в одной строке, не вынес в переменную. Поправили: «промежуточное имя — это форма документации».
- **🆕 Забывает прокинуть параметр после рефакторинга** — после удаления `analyze_expenses` забыл передать `limit` в `filter_expenses(expenses)`. Sanity check «запусти и сравни» — это именно тот случай, где он спасает.
- **🆕 PEP 8 в keyword-аргументах** — писал `limit = limit, category = category` вместо `limit=limit, category=category`. Линтеры на этапе 3 поправят, но привычка — сразу.
- **🆕 Имена коллекций (множественное число)** — снова всплыло: `summary_of_category` вместо `category_summaries`. Тенденция: сначала мысль на русском, потом транслирует в имя. Английский плюрал нужно проговаривать сразу.
- **🆕 Кавычки в f-строках** — пишет одинарные внутри одинарных (`f'... {x['key']} ...'`). В Python 3.12 работает, в более старых версиях — синтаксическая ошибка. Альтернатива — чередовать или вынести в переменную.
- **🆕 camelCase из JS просачивается** — написал `isGoodInput` вместо `is_good_input`. PEP 8 для переменных и функций — `snake_case` (классы — `PascalCase`, константы — `UPPER_SNAKE_CASE`). Это рефлекс с фронта, проявляется в первые секунды написания, потом сам ловит. Линтер на этапе 3 будет ругаться.
- **🆕 Переиспользование переменной с разными типами** — в первой версии retry-loop писал `limit = input(...)` (str), потом `limit = int(limit)` (int) в той же переменной. Это нарушение «одна переменная — одна роль». В TypeScript так не получится, в Python тоже не надо. Привычка с динамической типизации JS, требует осознанного контроля.
- **🆕 Промежуточная переменная без необходимости** — написал `threshold_expense = int(input(...)); return threshold_expense` там, где хватало `return int(input(...))`. Понял разницу с `matched_total = sum(...)` (где переменная нужна, потому что используется дважды). Граница: промежуточное имя нужно, если оно используется ≥2 раз или существенно поясняет смысл.
- **🆕 «Скорее всего» в сообщениях об ошибках** — программа точно знает, что данные битые (она же сама поймала исключение). Не надо извиняться за свой код — говорить пользователю прямо: «Это не целое число». Сообщения об ошибках должны быть **actionable**.
- **🆕 `\n` внутри `print` плюс `print()` снаружи** — двойной перевод строки. `print()` сам добавляет `\n` в конце, поэтому `print('...\n')` это «строка + пустая строка». Если нужно пустое разделение — отдельный `print()`.
- **🆕 Несогласованная обработка пустоты** — после добавления `if not expenses` в начале каскада в первой версии оставил сводку по категориям наружу без обёртки. На пустом списке программа печатала «Расходы отсутствуют» + «=== Сводка ===» без содержания. Правило: «если решил обработать пустоту в одном месте, обработай везде, где данные используются».

## Заметки ментору

### Стиль обучения, который работает:
- **Я пишу код сам, ментор направляет наводящими вопросами** — это согласовано и работает отлично. **Не давать готовые решения**, только подсказки.
- **Параллели с JS/TS** — очень помогают, использовать активно. Особенно хорошо зашли: spread `{...obj}` ↔ `{**obj}`, rest `...args` ↔ `*args`, иммутабельность state в React ↔ чистые функции, `.filter()/.map()` ↔ comprehensions, `(x || 0)` ↔ `dict.get(x, 0)`.
- **Маленькие задачи** (20-40 минут на заход) — формат удобный.
- **Code review после каждой задачи** — обязателен, с объяснением "почему так лучше".
- **Хвалить за конкретные хорошие решения**, не за факт сдачи задачи.
- **Объяснять "почему", а не только "как"** — этот формат заходит, я задаю много вопросов.
- **Бонус-вопросы «ответь словами, без кода»** — отлично работают для архитектурных тем (когда оставлять маленькую функцию, чистая ли функция, и т.д.). Заставляют думать, а не писать код по шаблону.
- **Признавать косяки в задании** — если ментор сформулировал задачу неполно/неточно, признать и поправить. Доверие к процессу важнее лица ментора.
- **🆕 Ловушки-вопросы перед решением** — «что произойдёт, если ...?». Когда ментор подсунул `{expense['category']: expense['amount'] for expense in expenses}` и спросил «что получится?», ученик сам разобрал проблему дубликатов ключей. Сильно лучше, чем «сейчас покажу типичную ошибку».
- **🆕 Возврат к ранее пройденным принципам в новом контексте** — DRY, SSOT, sanity check работают как «крючки», на которые можно вешать новые задачи. Каждое замечание привязывать к уже знакомому принципу, а не вводить новый каждый раз.
- **🆕 Эксперименты в IDE — лучший закрепитель** — ученик сам пошёл проверять `naked except` против Ctrl+C, увидел вживую неубиваемую программу. Это запомнится навсегда, в отличие от слов ментора. Если есть возможность «подталкивать к эксперименту, а не объяснять теорией» — использовать.
- **🆕 Концепция → синтаксис, а не наоборот** — в теме try/except сначала разобрали границы доверия, EAFP/LBYL, «когда ловить — когда нет», и **только потом** дали синтаксис `try/except`. Синтаксис впитался за минуты, потому что было понятно, **зачем** он нужен. Это работает.
- **🆕 Карта поломок как инструмент сканирования** — на каждой задаче спрашивать «какие строки могут упасть и почему» — отдельно от написания кода. Это тренирует тот навык, который у джунов часто провисает (edge cases).

### Что НЕ работает / чего избегать:
- Не вываливать сразу всю теорию — разбивать на куски.
- Не давать длинные решения сразу — сначала вопросы.
- Не возвращаться специально к олимпиадным алгоритмам — для моей цели (backend-контроль) это не критично.
- Не торопить с дедлайнами — у меня семья и неровный график, могу пропасть на неделю.
- Не давать абстрактные формулировки в задачах с рефакторингом — конкретно «один проход = один цикл по полному списку, без вызова других функций, которые тоже итерируют» лучше, чем «делай за один проход». На примерах с регрессией убедились.
- **🆕 Не перегружать одну сессию** — comprehensions вышло на 12-15 задач, под конец появлялись ошибки от усталости (опечатки, забытые параметры). Лучше закрывать чат после 8-10 итераций, даже если тема не закончена.

### Важные психологические моменты:
- **Боюсь забыть фронт** — иногда нужно напоминать, что fullstack > специалист в одном направлении, и что для забывания фронта нужны месяцы.
- **Боюсь забыть Python** — постоянное чувство "завтра всё забуду". Это нормально, нужно периодически фиксировать что я уже точно умею.
- **Семья на первом месте** — это правильно, не давить графиком.
- **🆕 Параллельные пет-проекты на фронте — это нормально**, не давить виной за «отвлёкся от Python на dnd-kit». Это часть работы fullstack-разработчика.

### Принципы, которые уже проработаны и работают как "крючки":
1. **"Код не знает данные заранее"** (обобщённость) — главный принцип, разобран на примере с категориями расходов.
2. **"Данные отдельно, представление отдельно"** — separation of concerns.
3. **"Сначала собери, потом покажи"** — backend-паттерн, разобран на задаче с фильтрацией.
4. **"Множественное число для коллекций"** (`expenses`, не `expense_list`).
5. **"Деньги ≠ float"** — на будущее (`Decimal`).
6. **"Premature optimization is the root of all evil"** — читаемость > производительность на старте.
7. **"Имя функции = контракт"** — `format_*` обещает чистоту, `print_*`/`output_*` обещает побочный эффект. Имя должно честно описывать, что функция делает.
8. **"Functional core, imperative shell"** — грязно по краям (input/print/БД/HTTP), чисто внутри (расчёты/трансформации/фильтры). Базовая архитектура backend-кода.
9. **"Чистая функция = детерминированный вход→выход + ноль побочных эффектов"** — два критерия. Композиция чистых функций → чистая функция.
10. **"Дефолт для list/dict/set — это `None`"** — иначе словишь mutable default arguments. Внутри функции `if x is None: x = []`.
11. **"DRY: дубль условий → объединяй через and/or"** — две ветки `if`/`elif` с одинаковым `append` — это одно условие, неправильно сформулированное.
12. **"Маленькие функции + большие сборки"** — `filter_expenses` и `analyze_expenses` сосуществуют. Не выбор «или-или», а композиция.
13. **"Sanity check после рефакторинга"** — запусти и сравни с тем, что было до. Не доверяй коду «потому что компилируется».
14. **"Кортеж создаётся запятой, а не скобками"** — `1, 2, 3` это уже tuple. Скобки только для группировки/читаемости.
15. **"Нейтральный элемент аккумулятора"** — 0 для суммы, 1 для произведения, "" для строк, [] для списков, True для and, False для or. Гарантирует осмысленный результат на пустом входе.
16. **🆕 "Comprehension не пишет логику, он применяет"** — если вычисление уже в функции, comprehension вызывает функцию, а не дублирует формулу. `[format_expense(e) for e in expenses]`, не `[f'- {e["category"]}, ...' for e in expenses]`.
17. **🆕 "SSOT (Single Source of Truth) для дефолтов"** — дефолтное значение параметра живёт в одном месте, обычно во внешней функции. Внутренние помощники получают значение явно, без своего дефолта. Иначе дефолты разойдутся при изменении.
18. **🆕 "Промежуточное имя = форма документации"** — `matched_total = sum(...)` лучше, чем повторение `sum(...)` в строке. Имя объясняет «что это», выражение — «как посчитано».
19. **🆕 "Имя функции — измерение группировки"** — `group_by_category`, не `group_by_expenses`. Группируют **по чему**, а не **что** (что — понятно из аргументов).
20. **🆕 "Comprehension ≠ комбайн"** — если цикл делает несколько вещей одновременно (собирает + считает + аккумулирует), comprehension не подходит. Один comprehension = одна коллекция на выходе.
21. **🆕 "Trade-off: один проход vs читаемость"** — для маленьких/средних данных три прохода через `sum()`/`filter` читабельнее одного цикла с тремя аккумуляторами. Решение зависит от размера данных и того, где узкое место (в backend часто узкое место — БД, не Python).
22. **🆕 "GROUP BY-паттерн"** — `dict.get(key, 0) + amount` это аналог SQL `GROUP BY key, SUM(amount)`. Базовый паттерн агрегации в Python.
23. **🆕 "Удалённый код = удалённый код"** — после рефакторинга мёртвые функции удаляются, а не комментируются. Git помнит всё.
24. **🆕 "Исключения — это второй канал связи"** — `return` говорит «вот результат», `raise` говорит «я не смог, вот почему». Обрабатывает тот уровень, который **знает**, что делать с ошибкой.
25. **🆕 "EAFP vs LBYL"** — EAFP (try/except) для непредсказуемых ошибок извне (битый ввод, сломанный файл, упавшая сеть). LBYL (`if ...`) для предсказуемых бизнес-условий на своих данных. **EAFP — питонячий стиль по умолчанию.**
26. **🆕 "Не лови, если не знаешь что делать"** — молчаливо проглоченное исключение хуже, чем падение. Если нет конкретного ответа на «что делать с ошибкой?» — не лови, пробрасывай наверх.
27. **🆕 "Цель `except` — вернуть в рабочее состояние, а не заглушить"** — retry-loop, fallback на дефолт, понятное сообщение пользователю — это работа. `except: pass` — антипаттерн.
28. **🆕 "Граница доверия"** — данные из `input`, файлов, сети — опасны (нужны проверки/обработка ошибок). Данные изнутри программы (литералы, результаты своих функций) — безопасны. Защита нужна **на границе**, не везде.
29. **🆕 "Проверяй причину, а не следствие"** — `if not expenses` (причина: список пуст) лучше, чем `if all_total == 0` (следствие: сумма ноль). Следствие может совпасть случайно при других причинах (например, разнознаковые суммы).
30. **🆕 "Naked `except:` — антипаттерн"** — ловит вообще всё, включая `KeyboardInterrupt` (Ctrl+C) и `SystemExit`. Программа становится неубиваемой через клавиатуру. Всегда указывай конкретный тип. `except Exception:` допустимо — оно не ловит служебные исключения.
31. **🆕 "Trace-back читается снизу вверх"** — внизу тип ошибки и место, где она случилась. Выше — путь вызовов («most recent call last»). При дебаге backend-кода в FastAPI/Django стек будет 10-20 кадров глубиной, привычка читать снизу — обязательна.
32. **🆕 "Согласованность обработки пустоты"** — если решил, что «X может быть пустым» в одной точке кода — это решение должно распространиться **на весь код**, который работает с X. Защита в одной точке + предположение «непустого» в другой = противоречие, которое выстрелит позже.
33. **🆕 "Одна переменная — одна роль"** — не пиши `limit = input(...)` (str), потом `limit = int(limit)` (int) в одной переменной. Имя должно соответствовать **одному** типу/смыслу на протяжении всей жизни. TS-аналог: `let limit: string | number` — так не делают.
34. **🆕 "Сообщения об ошибках должны быть actionable"** — не «произошла ошибка», а «это не целое число, попробуйте ещё раз». Из сообщения должно быть понятно, **что именно сделать иначе**. Не извиняйся за свой код — программа знает, что не так.
35. **🆕 "Try должен накрывать минимум кода"** — только тот, что реально может бросить ожидаемое исключение. Остальное — в `else:`. Иначе случайно поймаешь чужой `ValueError` и неправильно его интерпретируешь.

### Технические детали:
- **Type hints обязательны на параметрах и возврате** — раз TS-фон, требуем сразу (`def f(x: int) -> str`). На локальных переменных — избыточны.
- **Docstrings приветствуются** — особенно `"""..."""` после `def`. После рефакторинга — обновлять синхронно.
- **PyCharm подсветки уважаем** — обращать внимание на жёлтые предупреждения IDE.
- **Запретные имена переменных** — `list`, `dict`, `str`, `int`, `id`, `type`, `sum` и т.д.
- **Современный синтаксис типов** — `int | float` вместо `Union[int, float]`, `str | None` вместо `Optional[str]`, `list[dict]` вместо `List[Dict]` (Python 3.10+ — у нас 3.12.10, всё доступно).
- **Конвенции порядка параметров в сигнатуре** — `(обычные, *args, keyword_only, **kwargs)` именно в таком порядке.
- **Вызов по имени для 3+ параметров** — `f(expenses, limit=100, category="еда")`, не `f(expenses, 100, "еда")`. На месте вызова читаемость > краткость.
- **Линтеры на этапе 3 уберут** мелкие проблемы со стилем (лишние пробелы, пустые строки) — не отвлекаться сейчас на это.
- **🆕 PEP 8 в keyword-аргументах: без пробелов вокруг `=`** — `f(x=1)`, не `f(x = 1)`. Это касается и сигнатур с дефолтами, и вызовов.
- **🆕 Кавычки в f-строках — чередовать** — `f'... {x["key"]} ...'` или `f"... {x['key']} ..."`. Одинаковые внутри-снаружи работают только с 3.12+. Лучшая практика — вынести значение в переменную, тогда вопрос отпадает.
- **🆕 `dict.get(key, default)`** — стандартный способ безопасно прочитать ключ с дефолтом. Аналог `(obj[key] || default)` в JS, но без багов на falsy-значениях.
- **🆕 `'\n'.join(list_of_strings)`** — стандартный способ напечатать список строк построчно. Обязательно знать.
- **🆕 PEP 8 имена**: переменные/функции — `snake_case`, классы — `PascalCase`, константы — `UPPER_SNAKE_CASE`. Никаких `camelCase` из JS.
- **🆕 `print()` сам добавляет `\n`** — `print('text\n')` даёт двойной перенос. Для пустой строки — отдельный `print()`.
- **🆕 Иерархия исключений**: `BaseException > {KeyboardInterrupt, SystemExit, Exception} > {ValueError, KeyError, ZeroDivisionError, ...}`. Лови `Exception` или конкретные типы, **никогда** голый `except:`.
- **🆕 Имена исключений = описание проблемы**: `ValueError` (плохое значение), `TypeError` (плохой тип), `KeyError` (нет ключа), `IndexError` (нет индекса), `ZeroDivisionError`, `FileNotFoundError`, `AttributeError`.

## Финальный код v1.1 проекта (после темы try/except)

```python
def format_expense(expense: dict) -> str:
    """Форматирует расход в строку для вывода."""
    return f'- {expense["category"]}, {expense["amount"]} руб, {expense["description"]}'


def matches_filters(expense: dict, *, limit: int | float, category: str | None) -> bool:
    """Проверяет соответствие расхода категории и лимита."""
    return (category is None or expense['category'] == category) and expense['amount'] >= limit


def filter_expenses(expenses: list, *, limit: int | float = 0, category: str | None = None) -> list:
    """Возвращает расходы с суммой не меньше limit и указанной category."""
    return [
        expense for expense in expenses
        if matches_filters(expense, limit=limit, category=category)
    ]


def group_by_category(expenses: list) -> dict[str, float | int]:
    """Группирует расходы по категории, считая сумму в каждой."""
    totals = {}
    for expense in expenses:
        category = expense['category']
        amount = expense['amount']
        totals[category] = totals.get(category, 0) + amount
    return totals


def read_limit() -> int:
    """Запрашивает у пользователя порог в рублях, повторяя при некорректном вводе."""
    while True:
        try:
            return int(input('Введите порог в рублях: '))
        except ValueError:
            print('Это не целое число. Попробуйте ещё раз.')
            print()


# === Imperative shell ===
limit = read_limit()

expenses = [
    {'category': 'еда', 'amount': 500, 'description': 'обед в кафе'},
    {'category': 'транспорт', 'amount': 60, 'description': 'метро'},
    {'category': 'развлечения', 'amount': 1500, 'description': 'кино'},
    {'category': 'еда', 'amount': 1200, 'description': 'ужин'},
    {'category': 'транспорт', 'amount': 350, 'description': 'такси'},
    {'category': 'еда', 'amount': 200, 'description': 'кофе'},
]

matched_expenses = filter_expenses(expenses, limit=limit)

if not expenses:
    print('Расходы отсутствуют')
elif not matched_expenses:
    print('Расходов выше порога не найдено')
else:
    matched_total = sum(expense['amount'] for expense in matched_expenses)
    all_total = sum(expense['amount'] for expense in expenses)
    print(f'Расходы от {limit} руб:')
    formatted_expenses = [format_expense(expense) for expense in matched_expenses]
    print('\n'.join(formatted_expenses))
    print()
    print(f'Найдено: {len(matched_expenses)} шт.')
    print(f'Общая сумма: {matched_total} руб')
    print(f'Выше порога: {(matched_total * 100 / all_total):.2f}% от всех трат')

if expenses:
    print()
    print('=== Сводка по категориям ===')
    totals = group_by_category(expenses)
    category_summaries = [f'- {category}: {amount} руб' for category, amount in totals.items()]
    print('\n'.join(category_summaries))
```

**Изменения по сравнению с v1.0:**
