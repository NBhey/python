### Работа с файлами — детализация пройденного: 🆕
- [x] **Концепция «open → read/write → close»** — файл это поток байтов на диске, а не объект в памяти. Аналогия с библиотекой: взял книгу, почитал, **вернул обратно**. Иначе утечка ресурса.
- [x] **Что бывает, если не закрыть**: утечка file descriptor (лимит ОС ~1024), буферизованные данные не долетают до диска, файл блокирован на Windows.
- [x] **`open(path, mode, encoding='utf-8')`** — три параметра, кодировка обязательна явно.
- [x] **Режимы**: `'r'` (чтение, файла нет → `FileNotFoundError`), `'w'` (запись, создаёт/**стирает**), `'a'` (дозапись, создаёт), `'x'` (создать новый, упасть если есть), `'r+'` (читать+писать).
- [x] **`'w'` стирает файл в момент `open`**, а не в момент `.write()` — типичный источник трагедий «куда делись данные?!»
- [x] **`encoding='utf-8'` — всегда явно**. Иначе на Windows возьмётся `cp1251` и русский потеряется.
- [x] **`open` возвращает file object** — «ручку» через которую работаем.
- [x] **Контекстный менеджер `with` — единственно правильный способ работать с файлами**. Гарантированный close даже при исключении. Аналог `try/finally`, встроенный в синтаксис.
- [x] **Несколько `with` в одной строке**: `with open(...) as a, open(...) as b:` (или круглые скобки в 3.10+). Для копирования/трансформации между двумя файлами.
- [x] **Чтение**: `.read()` (всё), `.readlines()` (список строк), **`for line in file:`** (питонячий — ленивая итерация, не грузит весь файл в память).
- [x] **Запись**: `.write(s)` (без авто-`\n`!), `.writelines(list)`.
- [x] **`enumerate(file, start=1)`** — питонячий способ нумерации, работает на любом итерируемом, не только списке.
- [x] **`.rstrip()`** на прочитанной строке — убирает `\n` с конца.
- [x] **Карта поломок файлов**: `FileNotFoundError`, `PermissionError`, `OSError`, `UnicodeDecodeError` (битая кодировка).
- [x] **EAFP по умолчанию для файлов** — `try/except FileNotFoundError`, не `if Path(...).exists()`. Причина: TOCTOU race condition в LBYL-варианте.
- [x] **Защищай чтение, а не запись** — `'w'`/`'a'`/`'x'` сами создают файл, `FileNotFoundError` на записи невозможен.
- [x] **«Истина в памяти, файл — снимок»** — для вывода берём данные из памяти, не перечитываем файл после сохранения.

### JSON — детализация пройденного: 🆕
- [x] **Сериализация vs десериализация** — структура → строка / строка → структура. Базовый термин backend.
- [x] **Почему не `eval(str(obj))`**: arbitrary code execution через десериализацию. Тот же риск с `pickle` — не использовать для данных извне.
- [x] **JSON — формат без кода, только данные**, понимается всеми языками. На бэке 99% обмена с фронтом — JSON.
- [x] **`dumps`/`loads` ↔ строки, `dump`/`load` ↔ файлы**. Мнемоника: `s` от **s**tring.
- [x] **Параллель с JS**: `JSON.stringify` ↔ `json.dumps`, `JSON.parse` ↔ `json.loads`. Один концепт, два языка.
- [x] **`ensure_ascii=False, indent=2`** — рабочий комбо для записи. Без `ensure_ascii=False` русский становится `\u0435\u0434\u0430` — валидно, но нечитаемо глазами.
- [x] **Соответствие типов**: `dict`↔object, `list/tuple`↔array, `str`↔string, `int/float`↔number, `True/False`↔`true/false`, `None`↔`null`.
- [x] **Подводные камни round-trip**: ключи dict после load всегда `str` (числовые ключи теряются); `tuple` → `list`; `datetime/Decimal/set/кастомные классы` не сериализуются из коробки.
- [x] **`json.JSONDecodeError`** — наследуется от `ValueError`. Кидается, когда строка не парсится как JSON.
- [x] **«JSON распарсился» ≠ «структура правильная»** — два разных слоя проверки. `json.load` отвечает только за синтаксис. Валидация формы (это dict, есть ключ `category`, amount это число) — это **отдельная задача**, на этапе 4-5 будет Pydantic.
- [x] **Защита загрузки**: ловим `(FileNotFoundError, json.JSONDecodeError)`, возвращаем разумный дефолт (`[]` или `{}`).
- [x] **Параллель с `localStorage`**: `getItem(...) ?? []` это та же идея «загрузить состояние с дефолтом», только LBYL-стиль через `null`, тогда как Python — EAFP через исключение.

### Архитектурные принципы файлов/JSON: 🆕
- [x] **«Состояние vs логи»**: `'w'` для редактируемого состояния (расходы, конфиг, корзина), `'a'` для append-only логов (access.log, журнал транзакций). **БД — когда вырос настолько, что файл целиком уже не годится** (этап 4).
- [x] **Граница перехода файл → БД** ≈ «миллионы записей и/или редактирование в конкурентной среде». До этого — JSON-файл это правильный инструмент, не «костыль».
- [x] **Пиши только при мутации** — в `report` ничего не меняется, `save` не нужен. В `add` мутация → `save` обязателен. Не дёргать диск зря.

- **Docstring отстаёт от кода** — после рефакторинга функция меняется, а docstring остаётся
- **Опечатки в русских docstrings** — мелкие, но регулярные
- **Перенос логики при рефакторинге** — болячка из v1.0, всплывает снова
- **Регрессии при рефакторинге сигнатуры** — болячка возвращается
- **Дублирование вычислений** — иногда забывает вынести в переменную
- **PEP 8 в keyword-аргументах и type hints** — пробелы вокруг `=`, после `:` и `,`. **🆕 Снова всплыло в v2.0** (`path:str` вместо `path: str`, `dict[str,str]` вместо `dict[str, str]`). Заметка: упомянуть в начале каждой новой темы пока не закрепится.
- **Имена коллекций (множественное число)** — заметно лучше, но иногда срывается
- **Кавычки в f-строках** — пишет одинарные внутри одинарных
- **camelCase из JS просачивается** — обычно ловит сам после написания
- **Переиспользование переменной с разными типами** — болячка с JS
- **Промежуточная переменная без необходимости** — иногда лишняя
- **«Скорее всего» в сообщениях об ошибках** — закрыто
- **`\n` внутри `print` плюс `print()` снаружи** — иногда двойной перевод
- **Несогласованная обработка пустоты** — болячка возвращается при рефакторинге. **🆕 В v2.0 повторилось:** `show_expenses_report` потерял трёхуровневый каскад из v1.1, остался только `if not matched_expenses`. На пустом списке программа пишет «расходов выше порога не найдено» и показывает пустую сводку. Та же ошибка по той же причине, что в v1.0. Это **системная регрессия при выделении функций** — стоит сделать чек-листом перед закрытием функции: «что эта функция увидит на пустом входе?».
- **🆕 Type alias именует структуру, а не роль** — `Expense = dict[str, float | int]` некорректен, потому что в расходе значения **разных типов** (`str` для category, `int` для amount). PyCharm должен ругаться. Для словарей с разными типами полей правильный инструмент — `TypedDict` (этап 2). Пока: в сомнении — не делать alias, оставлять `dict`.
- **🆕 Один alias для двух разных ролей** — попытался назвать и расход (`dict[...]`), и результат `group_by_category` одним именем `Expense`. Принцип: **alias именует роль, а не форму**. Если два словаря случайно похожи по форме, это не повод объединять.
- **🆕 Не вернул `read_limit` в v2.0** — выделил функцию в v1.1, потом при рефакторинге в v2.0 написал голый `int(input(...))` в `show_expenses_report`. Регрессия по защите от битого ввода. **Болячка той же природы, что несогласованная обработка пустоты** — после рефакторинга теряет ранее установленные защиты.
- **🆕 `json.decoder.JSONDecodeError` vs `json.JSONDecodeError`** — пишет длинную форму, лезет в подмодуль. Короткая официальна. «Не лезь в кишки модуля без нужды».

33. **"Одна переменная — одна роль"**
34. **"Сообщения об ошибках должны быть actionable"**
35. **"Try должен накрывать минимум кода"**
36. **🆕 "`with` — единственно правильный способ работы с файлами"** — гарантирует close, аналог try/finally встроенный в синтаксис. На этапе 2 будем писать свои контекстные менеджеры.
37. **🆕 "Защищай чтение, а не запись"** — `'w'`/`'a'`/`'x'` сами создают файл. `FileNotFoundError` возможен только на `'r'`. Не лови «на всякий случай».
38. **🆕 "EAFP по умолчанию для файлов"** — `try/except FileNotFoundError`, не `if Path.exists()`. Причина: TOCTOU race condition в LBYL-варианте.
39. **🆕 "Состояние vs логи"** — `'w'` для редактируемых данных, `'a'` для append-only. БД — когда вырос.
40. **🆕 "Пиши только при мутации"** — диск не дёргать ради чтения. Save после `add`, не после `report`.
41. **🆕 "Истина в памяти, файл — снимок"** — после load работаем с in-memory структурой, save это её слепок, для вывода не перечитывать.
42. **🆕 "Сериализация = структура → строка, десериализация = строка → структура"** — базовый термин backend, везде будет всплывать.
43. **🆕 "JSON-формат без кода"** — поэтому безопасен в отличие от `eval(str(obj))` или `pickle`. Не использовать pickle на данных извне.
44. **🆕 "`dumps`/`loads` — строки, `dump`/`load` — файлы"** — мнемоника `s` от **s**tring.
45. **🆕 "`ensure_ascii=False` — для людей, не для парсеров"** — без него русский становится `\uXXXX`, валидно но нечитаемо.
46. **🆕 "JSON распарсился ≠ структура правильная"** — два разных слоя. Парсер отвечает только за синтаксис, форму надо проверять отдельно. Pydantic решает на этапе 4-5.
47. **🆕 "Alias именует роль, а не форму"** — если две структуры случайно похожи, не объединять в один alias.
48. **🆕 "Инструмент по размеру задачи"** — JSON-файл норм до десятков тысяч записей, потом БД. Не подключать PostgreSQL для 100 строк, не использовать JSON для биржи.

### Технические детали:
- **Type hints обязательны на параметрах и возврате**, на локальных переменных — избыточны
- **Docstrings приветствуются**, обновлять синхронно с рефакторингом
- **PyCharm подсветки уважаем**
- **Запретные имена переменных** — `list`, `dict`, `str`, `int`, `id`, `type`, `sum`, и т.д.
- **Современный синтаксис типов** — `int | float`, `str | None`, `list[dict]`
- **Конвенции порядка параметров** — `(обычные, *args, keyword_only, **kwargs)`
- **Вызов по имени для 3+ параметров**
- **Линтеры на этапе 3 уберут** мелочи стиля
- **PEP 8 в keyword-аргументах: без пробелов вокруг `=`**
- **Кавычки в f-строках — чередовать**
- **`dict.get(key, default)`** — стандарт
- **`'\n'.join(list_of_strings)`** — стандарт для печати списка построчно
- **PEP 8 имена**: `snake_case` для переменных/функций, `PascalCase` для классов, `UPPER_SNAKE_CASE` для констант
- **`print()` сам добавляет `\n`**
- **Иерархия исключений**: `BaseException > {KeyboardInterrupt, SystemExit, Exception} > {ValueError, KeyError, ZeroDivisionError, ...}`
- **Имена исключений = описание проблемы**
- **🆕 `with open(path, mode, encoding='utf-8') as file:` — мантра.** `encoding='utf-8'` всегда явно.
- **🆕 Режимы `open`**: `'r'` (читать, файла нет → падает), `'w'` (создать/стереть и писать), `'a'` (создать/писать в конец), `'x'` (создать только если нет), `'r+'`/`'w+'`/`'a+'` (комбинированные, в учебных задачах не нужны).
- **🆕 Чтение файла**: `for line in file:` + `.rstrip()` — питонячий по умолчанию. `.read()` и `.readlines()` — только при необходимости.
- **🆕 `enumerate(iterable, start=1)`** — работает на любом итерируемом, не только списке. На файле тоже.
- **🆕 `json.dump(obj, file, ensure_ascii=False, indent=2)`** — стандартный вызов для записи в файл.
- **🆕 `json.load(file)`** — стандартный вызов для чтения из файла.
- **🆕 `json.JSONDecodeError`** — короткая форма, не `json.decoder.JSONDecodeError`.
- **🆕 Карта обработки загрузки JSON-файла**: `try: open + json.load; except FileNotFoundError: дефолт; except json.JSONDecodeError: дефолт + предупреждение`.
- **🆕 `TypedDict` для словарей с разными типами полей** — `from typing import TypedDict`. Аналог `interface` в TS. **Разберём подробно на этапе 2.**

## Финальный код v2.0 проекта (после темы файлов/JSON)

```python
import json

FILE_PATH = 'expenses.json'


def format_expense(expense: dict) -> str:
    """Форматирует расход в строку для вывода."""
    return f'- {expense["category"]}, {expense["amount"]} руб, {expense["description"]}'


def matches_filters(expense: dict, *, limit: int | float, category: str | None) -> bool:
    """Проверяет соответствие расхода категории и лимиту."""
    return (category is None or expense['category'] == category) and expense['amount'] >= limit


def filter_expenses(expenses: list, *, limit: int | float = 0, category: str | None = None) -> list:
    """Возвращает расходы с суммой не меньше limit и указанной category."""
    return [
        expense for expense in expenses
        if matches_filters(expense, limit=limit, category=category)
    ]


def group_by_category(expenses: list) -> dict[str, int | float]:
    """Группирует расходы по категории, считая сумму в каждой."""
    totals = {}
    for expense in expenses:
        totals[expense['category']] = totals.get(expense['category'], 0) + expense['amount']
    return totals


def read_int(prompt: str) -> int:
    """Запрашивает у пользователя целое число, повторяя при некорректном вводе."""
    while True:
        try:
            return int(input(prompt))
        except ValueError:
            print('Это не целое число. Попробуйте ещё раз.')
            print()


def load_expenses(path: str) -> list:
    """Загружает список расходов. На отсутствие файла или битый JSON — пустой список."""
    try:
        with open(path, 'r', encoding='utf-8') as file:
            return json.load(file)
    except FileNotFoundError:
        print('Файл расходов не найден, начинаю с пустого списка.')
        return []
    except json.JSONDecodeError:
        print('Файл расходов повреждён, начинаю с пустого списка.')
        return []


def save_expenses(path: str, expenses: list) -> None:
    """Сохраняет список расходов в JSON-файл."""
    with open(path, 'w', encoding='utf-8') as file:
        json.dump(expenses, file, ensure_ascii=False, indent=2)


def create_new_expense() -> dict:
    """Запрашивает у пользователя данные для нового расхода."""
    category = input('Категория расхода: ').strip()
    amount = read_int('Потраченная сумма: ')
    description = input('Описание: ').strip()
    return {'category': category, 'amount': amount, 'description': description}


def show_expenses_report(expenses: list) -> None:
    """Печатает отчёт по расходам выше порога и сводку по категориям."""
    limit = read_int('Введите порог в рублях: ')
    matched_expenses = filter_expenses(expenses, limit=limit)

    if not matched_expenses:
        print('Расходов выше порога не найдено')
    else:
        matched_total = sum(expense['amount'] for expense in matched_expenses)
        all_total = sum(expense['amount'] for expense in expenses)
        print(f'Расходы от {limit} руб:')
        formatted_expenses = [format_expense(expense) for expense in matched_expenses]

    print('=== Сводка по категориям ===')
    totals = group_by_category(expenses)
    category_summaries = [f'- {category}: {amount} руб' for category, amount in totals.items()]
```
